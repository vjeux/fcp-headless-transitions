[[ visible ]] FragmentOut HgcZoomBlur_hgc_visible(const constant float4* hg_Params, 
    texture2d< float > hg_Texture0, 
    sampler hg_Sampler0,
    float4 texCoord0,
    float4 texCoord1,
    float4 texCoord2,
    float4 texCoord3,
    float4 texCoord4)
{
    const float4 c0 = float4(0.1500000060, 0.1000000015, 0.2000000030, 0.2500000000);
    const float4 c1 = float4(0.3000000119, 0.000000000, 0.000000000, 0.000000000);
    float4 r0, r1;
    FragmentOut output;

    r0 = hg_Texture0.sample(hg_Sampler0, texCoord1.xy);
    r0 = r0*c0.xxxx;
    r1 = hg_Texture0.sample(hg_Sampler0, texCoord2.xy);
    r1 = r1*c0.yyyy + r0;
    r0 = hg_Texture0.sample(hg_Sampler0, texCoord3.xy);
    r1 = r0*c0.zzzz + r1;
    r0 = hg_Texture0.sample(hg_Sampler0, texCoord4.xy);
    r0 = r0*c0.wwww + r1;
    r1 = hg_Texture0.sample(hg_Sampler0, texCoord0.xy);
    output.color0 = r1*c1.xxxx + r0;
    return output;
}



## ZoomBlur STRUCTURE DECODED (2026-07-23) — it's a 5-tap POLAR kernel, NOT a log-polar Gaussian
The verbatim HgcZoomBlur shader (above) is a fixed 5-TAP weighted blur:
  out = 0.30*sample(tc0) + 0.15*sample(tc1) + 0.10*sample(tc2) + 0.20*sample(tc3) + 0.25*sample(tc4)
  (weights c0=(0.15,0.10,0.20,0.25), c1.x=0.30; sum=1.0). The 5 texCoords tc0..tc4 are radial
  offsets set per-vertex/frameSetup.
PIPELINE (from -[PAEZoomBlur canThrowRenderOutput] @0x75718): rectToPolar(center, upscaleFactor)
  -> apply the 5-tap kernel IN POLAR SPACE (so the taps are RADIAL steps) -> polarToRect. Uses
  HGTransform::Scale + getInversePixelTransformForImage; getIntValue reads a pass/quality count and
  getFloatValue reads Amount + the X/Y centre (getXValue:YValue:).
CONSEQUENCE: the engine's current spatial.PAEZoomBlur model (a log-polar GAUSSIAN, directional-blur.ts)
  is STRUCTURALLY WRONG — FCP uses a small fixed 5-tap kernel in polar space, applied over N passes
  (the multi-pass accumulation is what spreads the bright disc radially "far more than the TS
  log-polar Gaussian" — the documented +12 inner / -21 mid signed residual at 14.5 dB). To decode:
  (1) read the tap radial offsets (the texCoord0..4 deltas) and the Amount->offset map + pass count
  from canThrowRenderOutput's texcoord/SetParameter math; (2) port rectToPolar/polarToRect with FCP's
  upscaleFactor + centre; (3) iterate the 5-tap kernel N passes. Gate-inert (shipping users author
  Amount=0). This REPLACES the "log-polar centre-resampling ceiling" characterization: the ceiling
  wasn't resampling precision, it was the WRONG KERNEL (Gaussian vs 5-tap-polar-multipass).

## CORRECTION 2026-07-23x — the previous "5-tap is the render path" claim was WRONG (retracted)
Verified against -[PAEZoomBlur canThrowRenderOutput] @0x75718: the render path binds
__ZN16HDirectionalBlur (HDirectionalBlur, at 0x75aa0/0x75aec) — the LOG-POLAR GAUSSIAN — NOT the
5-tap HgcZoomBlur shader. The 5-tap HgcZoomBlur_hgc_visible is the OSC PREVIEW / fast path (the
engine's directional-blur.ts note already documented this correctly; my commit 31a1b6d claimed the
opposite and was mistaken). So:
  * The engine's spatial.PAEZoomBlur model (log-polar remap -> 1-D Gaussian on ln(r) -> inverse) is
    STRUCTURALLY CORRECT and matches the render pipeline (rectToPolar/polarToRect + HDirectionalBlur).
  * The 14.5 dB on the off-centre hard-disc synth IS the documented log-polar CENTRE-RESAMPLING
    ceiling (u=ln r -> -inf as r->0; bilinear resampling loses fidelity at small r that no sigma tune
    recovers), NOT a wrong-kernel error. Closing it needs a super-sampled/centre-refined log-polar
    buffer (deep, near-zero gate ROI since shipping users author Amount=0).
LESSON REINFORCED (self-correcting): before claiming a shader IS the render path, confirm which
program the CPU render (canThrowRenderOutput) actually BINDS — many Hgc*_hgc_visible shaders are OSC
previews. Here the bind is HDirectionalBlur, confirming the existing log-polar decode.

## ZOOM_LOG_K probe 2026-07-23 (INCONCLUSIVE — recorded, no change made)
Tried the radial analog of the spin wedge: a single sharp centered RING edge (r=300), measure the
radial 10-90 width in ln(r) space vs Amount to decode ZOOM_LOG_K. RESULT: the ln-width was CONSTANT
(0.0433) across Amount=10/30/60 — i.e. the centered ring barely spreads even at Amount=60, giving
K≈0.27/0.089/0.044 (∝1/Amount, not a stable constant). This means either (a) a centered ring is a
poor probe for zoom blur (the radial displacement at these Amounts is small relative to the edge),
or (b) the Amount->radial-displacement law is much gentler than the sigma≈0.0075·Amount the current
ZOOM_LOG_K=1.0 assumes. Per decode-don't-fit, NO change made — the signal is too weak/ambiguous to
re-derive K. The rings-probe verification (24-29 dB) that fixed K=1.0 stands. ZoomBlur remains at its
documented log-polar centre-resampling ceiling; the off-centre-disc 14.5 dB is synth worst-case.
The clean-wedge technique that worked for the SPIN arc (angular) does not transfer cleanly to the
ZOOM radial axis with a centered ring — a proper zoom probe needs an off-centre feature at known
radius with a large-enough Amount to produce a measurable, non-clipping radial streak.
