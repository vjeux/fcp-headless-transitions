PAEBadTV frame-peak AGC — DECODED MECHANISM (2026-07-23, isolated rig)
================================================================================
Isolated rig: Waviness=Static=Roll=0, Saturate=0; vary the frame's bright content and read the
DARK-field center (far from any bright patch) through REAL headless FCP.

FINDING 1 — the "frame-peak AGC" IS a boost of the SCAN-LINE-BRIGHTNESS dark-band floor.
With Scan Line Brightness=1 (dark floor = 1.0 => scanF=mix(1,1,·)=1 everywhere), a bright patch
does NOT lift the dark field until it is huge (bf<=0.15 -> no lift, dark stays 40; bf=0.25 -> 94).
With the DEFAULT Scan Line Brightness=1.5, a TINY bright patch lifts the whole dark field:
  bf=0     -> 51
  bf=0.005..0.15 -> 123   (STEP: constant across two orders of magnitude of bright area)
  bf=0.25  -> 157
  bf>=0.5  -> 255
So the AGC scales the DARK-BAND FLOOR (Scan Line Brightness), not a separate global gain. If the
floor is already 1.0 (no dark bands), there is nothing to boost -> no AGC. This is why Scan Line
Brightness=1 is a clean rig for the desaturate/pointwise decode.

FINDING 2 — TRIGGER is a THRESHOLD on the frame MAX (peak), not the mean/area.
Fixed small patch (5% area), vary patch VALUE:
  patch<=128 -> NO lift (dark stays 51);  patch>=180 -> lift (123). Threshold in (128,180].
Area barely matters above a tiny floor (bf 0.005 already fully triggers) => it is a PEAK detector.

FINDING 3 — the boosted floor maps like a higher Scan Line Brightness.
Uniform dark dv=40, sweep Scan Line Brightness -> code factor (out/40):
  SLB 1.0/1.25/1.5/2.0/3.0/4.0/5.0 -> 40/46/51/62/84/106/128  =>  factor ≈ 0.5488*SLB + 0.4547
The triggered dark field (dv=40 -> 123, factor 3.075) corresponds to an EFFECTIVE Scan Line
Brightness ≈ 4.77. i.e. when a >~128-peak pixel appears anywhere, FCP boosts the dark-band floor
from the authored 1.5 to ≈4.8 (a fixed boosted level for a saturated 255 peak); bigger/brighter
frames push it further (bf=0.25 -> 157 => higher still; full white -> 255 => floor saturates).

## CORRECTION 2026-07-23v — the "frame-peak AGC" IS the CoreGraphics readback artifact (NOT an FCP effect)
Re-ran the bright-patch sweep with OZ_CLAMP_UNIT (per-channel [0,1] clamp before the CoreGraphics
ExtendedLinearSRGB->sRGB PNG write). RESULT:
  bright-frac   default-readback   OZ_CLAMP_UNIT
  0.0           51                 51
  0.005..0.15   123 (the "AGC")    51   <- AGC GONE
  0.25          157                103
  0.5..1.0      255                255
So the dark-field "lift" from a tiny bright patch is ENTIRELY the CoreGraphics per-IMAGE gamut
compression: when the frame contains ANY over-1.0 (>255-equivalent) pixel, CGImageDestination
applies a GLOBAL tone map that lifts EVERY pixel (confirmed earlier: CG's op is per-image, not
per-pixel, when over-1 values are present — see shared_clamp_overflow_analysis.txt 2026-07-23p).
The confusion arose because -[PAEBadTV canThrowRenderOutput] was disassembled and contains NO
frame-statistic / histogram / max-reduction call — it only reads params, builds the wavy/noise
tables, and the YCbCr matrix. There is no CPU AGC. The GPU HgcBadTV shader is per-pixel with no
frame stat either. The "AGC" was never in FCP — it is the shim's readback, exactly like the
"HGColorMatrix over-1.0 clamp". So BadTV has NO auto-gain: its true dark field is unlifted (51),
and the DIVERGED verdict is driven by (a) the Static 2-D mt19937 noise field (genuine stochastic)
and (b) any over-1 content triggering the CG readback in the whole-frame delta. The desaturate leg
is FIXED (Rec.709). The residual bf>=0.25 lift under OZ_CLAMP_UNIT (103, 255) is genuine white-area
saturation of the readback, not an effect. NET: two of BadTV's three "mystery" effects (over-1
clamp, frame-peak AGC) are the SAME CoreGraphics readback artifact; only the Static noise field is
a real un-ported effect. decode-don't-fit upheld — no curve was fitted; the artifact was isolated
at the source with OZ_CLAMP_UNIT + a binary disasm showing no CPU frame-stat.
