## Top three concrete inefficiencies

### 1. Pool relaunches a completed task when the commit subject omits the DONE keyword

`fct/swarm/pool.py:done_task_ids` matched a commit as completion only if the subject
contained `swarm|DONE|DROPPED|BLOCKED|NOOP`. T-G1's actual commit
`4e3c17a T-G1: Color_Planes 3D fold — driver offset shift for link source-reads`
lacks any of those keywords, so the pool never marked T-G1 done, refilled slot 4
with T-G1 → agent found the work already on main → NOCHANGE in
`T-G1.20260713-013622.log`. Pure waste, one agent-slot × ~1 min.

Test that reproduces the bug and validates the fix:

    $ python3 -c "import re; line='T-G1: Color_Planes 3D fold — driver offset shift';
                  print(bool(re.match(r'^(?:swarm\s+)?(T-[A-Za-z0-9]+)\b', line) and
                             re.search(r'\b(DONE|DROPPED|BLOCKED|NOOP)\b', line)))"
    False   # old regex misses T-G1

**Fix.** Widen the completion regex to accept `T-<id>:` (colon at end of the id
token) as *also* completion, in addition to the keyword form:

    ^(?:swarm\s+)?(T-[A-Za-z0-9]+)(?::|\s+(?:DONE|DROPPED|BLOCKED|NOOP)\b)

Verified against real commits: T-A3 DROPPED:, T-D1 DONE:, T-G1:, swarm reflect:
(swarm-reflect doesn't match — correct, `reflect` isn't a `T-*` task id).

### 2. push_helper.sh silently loses agents' work under the sandbox

push_helper.sh (added at cf72a11) was the intended fix for the `.git/worktrees/*`
sandbox block, but its captured-diff step is:

    cd "$WT"
    git add -A >/dev/null 2>&1 || true      # <-- sandbox blocks index.lock write
    git diff --cached origin/main > "$DIFF" # <-- reads empty index → empty diff
    [ ! -s "$DIFF" ] && { echo "nothing to push"; exit 0; }

Because the sandbox denies `.git/worktrees/<id>/index.lock` creation, `git add`
fails silently (the `|| true` swallows it), the index stays at pristine `origin/main`,
the diff is empty, and push_helper reports success while the agent's real work
never leaves disk. Verified in this session: `xattr -l /Users/vjeux/fct-swarm/frames/reflect/engine/`
shows `com.apple.provenance` — the same block extends to the frames dir, and
`touch /Users/vjeux/fct-swarm/frames/reflect/engine/probe` returns "Operation not
permitted" from a Claude Code shell.

**Fix.** Replaced the git-add + git-diff-cached path with rsync of the worktree's
file state into the /tmp clone (excluding `.git`, `engine/node_modules`, `venv`,
`.swarm_*`, `.fctcache/`), then `git add -A` + gate + commit + push *inside the
/tmp clone*. The /tmp clone has clean provenance so all git ops succeed. This also
captures locally-committed work if the agent got a few commits through before the
sandbox tightened, since rsync always reflects the current working tree.

### 3. Agents spent decode-time on already-resolved tasks

Six T-A3 runs each ran full census over 65 built-ins to independently re-derive
"gravity is on Particle Cells, not layers" and then exited NOCHANGE. Even though
the pool completion bug (#1) was the *root cause*, the agent itself did no
zeroth-step check to notice T-A3's commit was already on origin/main. Same
pattern: the T-G1 relaunched agent spent time verifying before printing NOCHANGE.

**Fix.** Added a "Zeroth step" to `agent_brief.md`: two grep-line checks (against
`git log origin/main` for a matching commit, and against `origin/main:ROADMAP.md`
for a DONE/DROPPED row). Either match → immediate `SWARM_RESULT NOCHANGE` + exit,
no census. Cost of the check: two `git` calls. Savings: whole agent lifetime × N
re-launches when the pool has a hiccup.

## Secondary fix: parse_tasks reads from origin/main

The pool used to `open(MAIN/ROADMAP.md).read()` — the local working-tree copy.
Nothing pulls MAIN, so a `DONE` mark landed by an agent's push was invisible to
the scheduler until someone manually pulled. Changed to
`git show origin/main:ROADMAP.md` so a fresh `git fetch` (which the pool already
does) is enough to see the current row status. Defensive belt-and-suspenders for
issue #1: even without the regex fix, this alone would have caught T-G1.

should also re-render the affected slug into the shared frames dir. Or the swarm
setup should render each newly-baselined slug once into shared frames before
seeding worktrees off it.
